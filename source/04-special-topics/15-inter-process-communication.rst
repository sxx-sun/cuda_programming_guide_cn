4.15. 进程间通信
================

由不同主机进程管理的多个 GPU 之间的通信，通过使用进程间通信（IPC）API 和 IPC 可共享内存缓冲区来支持，方法是创建进程可移植句柄，随后用于获取对等 GPU 上设备内存的进程本地设备指针。

同一进程内的任何主机线程创建的设备内存指针或事件句柄都可以被该进程内的任何其他线程直接引用。但是，设备指针或事件句柄在创建它们的进程之外无效，因此不能被属于不同进程的线程直接引用。要跨进程访问设备内存和 CUDA 事件，应用程序必须使用 CUDA 进程间通信（IPC）或虚拟内存管理 API 来创建进程可移植句柄，这些句柄可以使用标准主机操作系统 IPC 机制（例如进程间共享内存或文件）与其他进程共享。一旦进程可移植句柄在进程之间交换，就必须使用 CUDA IPC 或 VMM API 从句柄获取进程本地设备指针。然后，进程本地设备指针可以像在单个进程中一样使用。

用于单节点和单操作系统实例内 IPC 的相同类型的可移植句柄方法，也用于多节点 NVLink 连接集群中 GPU 之间的点对点通信。在多节点情况下，通信的 GPU 由在每个集群节点上的独立操作系统实例中运行的进程管理，需要在操作系统实例级别之上进行额外的抽象。多节点点对点通信是通过在多节点 GPU 对等节点之间创建和交换所谓的「fabric」句柄，然后在参与进程和对应于多节点秩的操作系统实例内获取进程本地设备指针来实现的。

有关用于建立和交换进程可移植句柄以及节点和操作系统实例可移植句柄的具体 API，请参见下文（单节点 CUDA IPC）和 :ref:`virtual-memory-management`，这些句柄用于获取 GPU 通信的进程本地设备指针。

.. note::

   使用 CUDA IPC API 和虚拟内存管理（VMM）API 进行 IPC 时，各自具有优势和限制。

   CUDA IPC API 目前仅在 Linux 平台上支持。

   CUDA 虚拟内存管理 API 允许在内存分配时对对等可访问性和共享进行每分配控制，但需要使用 CUDA Driver API。

4.15.1. 使用传统进程间通信 API 的 IPC
-------------------------------------

要跨进程共享设备内存指针和事件，应用程序必须使用 CUDA 进程间通信 API，该 API 在参考手册中有详细描述。IPC API 允许应用程序使用 ``cudaIpcGetMemHandle()`` 获取给定设备内存指针的 IPC 句柄。CUDA IPC 句柄可以使用标准主机操作系统 IPC 机制（例如进程间共享内存或文件）传递给另一个进程。 ``cudaIpcOpenMemHandle()`` 使用 IPC 句柄检索有效的设备指针，该指针可以在其他进程内使用。事件句柄可以使用类似的入口点共享。

使用 IPC API 的一个示例是单个主进程生成一批输入数据，使数据可供多个辅助进程使用，而无需重新生成或复制。

.. note::

   IPC API 仅在 Linux 上支持。

   注意，IPC API 不支持 ``cudaMallocManaged`` 分配。

   使用 CUDA IPC 相互通信的应用程序应该使用相同的 CUDA 驱动程序和运行时进行编译、链接和运行。

   由 ``cudaMalloc()`` 进行的分配可能是从更大的内存块中子分配的，出于性能原因。在这种情况下，CUDA IPC API 将共享整个底层内存块，这可能导致其他子分配被共享，从而可能导致进程之间的信息泄露。为防止此行为，建议仅共享具有 2MiB 对齐大小的分配。

   在计算能力 7.x 及更高的 L4T 和嵌入式 Linux Tegra 设备上，仅支持 IPC 事件共享 API。IPC 内存共享 API 在 Tegra 平台上不支持。

4.15.2. 使用虚拟内存管理 API 的 IPC
-----------------------------------

CUDA 虚拟内存管理 API 允许创建 IPC 可共享内存分配，并且通过操作系统特定的 IPC 句柄数据结构支持多个操作系统。
